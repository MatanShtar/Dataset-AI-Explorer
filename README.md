# Applied Materials — Dataset Explorer with AI Insights

My submission for the Software Student Take-Home Exercise (Option A) for the GenAI Infrastructure team.

A web application for exploring CSV datasets: upload a file, browse it in a paginated table, and ask an AI assistant questions about the data in natural language.

## Features

- **CSV upload** — drag & drop anywhere on screen or click to browse, with thorough server-side validation (file type, size limit, encoding, malformed/empty files)
- **Data table** — paginated browsing (100 rows per page) with animated page transitions, sticky header, and numeric formatting
- **AI chat** — ask questions about the loaded dataset; answers are generated by Google Gemini with the dataset's schema, summary statistics, and a data sample as context, rendered as markdown (including tables), with token usage shown per answer

## Architecture

```
┌──────────────────┐         ┌───────────────────────────┐         ┌────────────┐
│  React frontend  │  HTTP   │      FastAPI backend      │  HTTPS  │   Google   │
│  (Vite)          ├────────▶│  upload / rows / ask      ├────────▶│   Gemini   │
│  src/api.js      │         │  pandas in-memory store   │         │            │
└──────────────────┘         └───────────────────────────┘         └────────────┘
```

- **Frontend** (`frontend/`) — React + Vite. All backend calls are centralized in `src/api.js`; `App.jsx` owns the dataset state and conditionally renders the table and chat panels.
- **Backend** (`backend/`) — FastAPI. Parses uploaded CSVs with pandas into an in-memory store, serves paginated rows, and orchestrates the LLM: it builds the prompt context (schema, dtypes, summary statistics, up to 100 sample rows) and keeps the API key server-side. The frontend never talks to Gemini directly.

## Getting Started

### Prerequisites

- Python 3.12+
- Node.js 18+
- A Gemini API key ([Google AI Studio](https://aistudio.google.com/apikey))

### Backend

```powershell
cd backend
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
```

Create `backend/.env`:

```
GEMINI_API_KEY=your-key-here
```

Run the server:

```powershell
python -m uvicorn main:app --reload
```

Interactive API docs are served at `http://127.0.0.1:8000/docs`.

### Frontend

```powershell
cd frontend
npm install
npm run dev
```

Open `http://localhost:5173`. Both servers must be running.

A small sample dataset (`test_data.csv`) is included at the repo root for a quick try.

## Deployment

The two parts deploy independently to any free-tier platform. Example setup — backend on [Render](https://render.com), frontend on [Vercel](https://vercel.com):

### Backend (Render)

1. Create a **Web Service** from this repo with root directory `backend`.
2. Build command: `pip install -r requirements.txt`
3. Start command: `uvicorn main:app --host 0.0.0.0 --port $PORT`
4. Set environment variables: `GEMINI_API_KEY`, and `ALLOWED_ORIGINS` = the frontend URL (e.g. `https://your-app.vercel.app`).

### Frontend (Vercel)

1. Import the repo and set the root directory to `frontend` (Vite is auto-detected).
2. Set the environment variable `VITE_API_BASE` to the backend URL (e.g. `https://your-api.onrender.com`).
3. Deploy — the build command (`npm run build`) and output directory (`dist`) are the defaults.

## API Overview

| Endpoint  | Method | Description |
|-----------|--------|-------------|
| `/health` | GET    | Liveness check |
| `/upload` | POST   | Upload and validate a CSV; stores it as the active dataset |
| `/rows`   | GET    | Paginated rows of the active dataset (`limit`, `offset`) |
| `/ask`    | POST   | Natural-language question about the active dataset, answered by Gemini |

## Design Notes

- **LLM orchestration lives in the backend.** The frontend never talks to Gemini directly — the backend builds the prompt context (schema, dtypes, summary statistics, up to 100 sample rows) and keeps the API key server-side.
- **In-memory dataset store.** A single active dataset is held in process memory — appropriate for a single-user exercise; swapping in a persistent store would only touch the storage class.
- **Single source of truth for API calls.** All frontend `fetch` logic is centralized in `src/api.js`, with AbortController-based timeouts so a dead backend surfaces as a clear error instead of a hung UI.
- **Design tokens.** All visual constants (colors, spacing, sizes, animation timings) live in `src/tokens.css`; component styles reference tokens only.
- **Configurable model.** Defaults to `gemini-3.5-flash`; override with `GEMINI_MODEL` in `backend/.env`.

## What I'd Do Next

- **Server-side filtering and search** — extend `GET /rows` with per-column filters and free-text search so the table scales beyond what the client holds in memory.
- **Session-scoped datasets** — replace the single in-memory dataset with per-session storage (SQLite or Redis) so concurrent users don't overwrite each other's data.
- **LLM-generated pandas queries** — for aggregation questions, have the model emit a pandas expression that runs against the full dataset instead of reasoning over a 100-row sample, making answers exact on large files.
- **Streaming chat responses** — stream Gemini's answer token-by-token (SSE) so long answers render progressively instead of after a wait.
- **Tests and CI** — pytest coverage for the upload validation paths and `/ask` prompt construction, plus a GitHub Actions workflow running them on every push.
