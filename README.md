# Applied Materials — Dataset Explorer with AI Insights

My submission for the Software Student Take-Home Exercise (Option A) for the GenAI Infrastructure team.

A web application for exploring CSV datasets: upload a file, browse it in a paginated table, and ask an AI assistant questions about the data in natural language.

## Features

- **CSV upload** — drag & drop anywhere on screen or click to browse, with thorough server-side validation (file type, size limit, encoding, malformed/empty files)
- **Data table** — paginated browsing with animated page transitions, sticky header, and numeric formatting
- **AI chat** — ask questions about the loaded dataset; answers are generated by Google Gemini with the dataset's schema, summary statistics, and a data sample as context, rendered as markdown (including tables), with token usage shown per answer

## Getting Started

### Prerequisites

- Python 3.12+
- Node.js 18+
- A Gemini API key ([Google AI Studio](https://aistudio.google.com/apikey))

### Backend

```powershell
cd backend
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
```

Create `backend/.env`:

```
GEMINI_API_KEY=your-key-here
```

Run the server:

```powershell
python -m uvicorn main:app --reload
```

Interactive API docs are served at `http://127.0.0.1:8000/docs`.

### Frontend

```powershell
cd frontend
npm install
npm run dev
```

Open `http://localhost:5173`. Both servers must be running.

A small sample dataset (`test_data.csv`) is included at the repo root for a quick try.

## API Overview

| Endpoint  | Method | Description |
|-----------|--------|-------------|
| `/health` | GET    | Liveness check |
| `/upload` | POST   | Upload and validate a CSV; stores it as the active dataset |
| `/rows`   | GET    | Paginated rows of the active dataset (`limit`, `offset`) |
| `/ask`    | POST   | Natural-language question about the active dataset, answered by Gemini |

## Design Notes

- **LLM orchestration lives in the backend.** The frontend never talks to Gemini directly — the backend builds the prompt context (schema, dtypes, summary statistics, up to 50 sample rows) and keeps the API key server-side.
- **In-memory dataset store.** A single active dataset is held in process memory — appropriate for a single-user exercise; swapping in a persistent store would only touch the storage class.
- **Single source of truth for API calls.** All frontend `fetch` logic is centralized in `src/api.js`, with AbortController-based timeouts so a dead backend surfaces as a clear error instead of a hung UI.
- **Design tokens.** All visual constants (colors, spacing, sizes, animation timings) live in `src/tokens.css`; component styles reference tokens only.
- **Configurable model.** Defaults to `gemini-3.5-flash`; override with `GEMINI_MODEL` in `backend/.env`.
